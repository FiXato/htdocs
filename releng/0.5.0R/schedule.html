openMSX 0.5.0 Release Process

Introduction

Here is a page for openMSX 0.5.0 release specific schedule. It is intended for openMSX release engineering team and openMSX developers. End-users or for more generic information about openMSX and its development, see openMSX web site.

Currently, no general release engineering page of openMSX is available.

Most of detailed discussion are made on #openMSX at irc.freenode.net .

The current release engineering TODO list is also available.

Schedule

Action Expected Actual Description Responsible Updated
Freeze features 2004-10-15(UTC)   After this, no more new features can be commited. Even for critical bug fix, all commits must be approved by Release Engineer(s). mth, wouter 2004-10-15(UTC)
Freeze documents 2004-10-15(UTC)   At this point, all documents must be ready for release. Any description for all new features, all changed must be done. Version number in documents should be 0.5.0-Release. After this, even for critical fix, all commits must be approved by Release Engineer(s). Manuel 2004-10-15(UTC)
Build 0.5.0-RC1 2004-10-15(UTC) 2004-10-15(UTC) Build 0.5.0-RC1 source tarball and win32 binary package and put on hidden space. This should be tested heavily by release engineers and core developers and core testers. mth, tcm1998 2004-10-15(UTC)
Build 0.5.0-RC2     Build 0.5.0-RC2 and test. The needness of this is depend on the result of 0.5.0-RC1. So, no expected day. mth, tcm1998  
Tagging tree for release     Tag the tree with RELEASE_0_5_0 . Even if this is done, still there may be some fixes untill actuall release is done. So, no one should treat as release at this point. mth  
Build 0.5.0-Release     Build 0.5.0 and upload to sourceforge. mth, tcm1998
